## 1. 现象诊断

### 最可能原因 1：scoring / image aggregation 失配

当前 A1 的核心打分链路是：先算 `normal_map`，再定义 `anomaly_map = 1 - normal_map`，上采样后做 `topk_mean` 得到 image score。也就是说，**sign、aggregation 位置、top-k 比例**都被写死了。

而 A2 的结果已经把问题暴露得很清楚：`defect-minus-normal` 提升了 pixel AUROC 到 `0.5728`，但 image AUROC 仍只有 `0.4617`；反过来 `normal-minus-defect` 把 image AUROC 拉到 `0.5462`，却把 pixel AUROC 拉低到 `0.4272`。这不是“完全没信号”，而是**同一套特征里，局部定位和整图排序用到的方向不一致**。这是当前最核心的问题。

这和近年的 CLIP-AD 很像。CLIP-AD 明确指出，直接做 anomaly map 时会出现“相反预测”和“无关高亮”，根源在于 CLIP 表征与 anomaly scoring 之间并不天然对齐，因此它专门引入 staged dual-path 和 multi-level feature surgery 来修正这个问题。A2 指标冲突，和它描述的症状非常接近。([arXiv][1])

### 最可能原因 2：prototype 设计过于粗糙

现在的 `feature_bank.py` 是把所有 support patch feature flatten 之后，直接求一个**单一均值 prototype**。这对 normal 和 defect 都一样。

这个设计非常省显存，但代价也很明显：它会把正常纹理的多模态结构、局部几何差异、背景变化都压到一个中心点里。对于 few-shot anomaly detection，这通常太粗。PatchCore 的核心思想就是**不要把正常 patch 压成一个点，而是保留代表性的 nominal patch memory bank 做最近邻匹配**；DictAS、ReMP-AD 进一步说明 few-shot 场景里“参考样本并不等价”，需要 dictionary / memory retrieval，而不是单中心；Sea-CLIP 则强调 anomaly 比较需要**语义对应的 patch matching**，不能只靠粗糙全局原型。当前的单均值 prototype，和这条文献主线是反着来的。([arXiv][2])

### 很可能原因 3：B 的 supervision 太弱，导致 optimization 塌缩

这条在代码里是直接可见的。`StageA1Dataset` 已经把 mask 读进来了，但 `run_stage_b.py` 训练时只拿 `image_logits` 做 image-level BCE；没有 pixel loss，没有 Dice/Focal，也没有任何利用 defect mask 的监督。 

与此同时，B 的 head 只是一个很小的 `1x1 conv -> ReLU -> Dropout -> 1x1 conv`，输入也只有三张 similarity map：`normal_map / defect_map / contrast_map`。在 support 图像极少、监督只到 image level 的条件下，学成近常数输出是完全可以预期的。 

这里我要明确标一下：**“B 之所以塌缩，主要是 supervision 太弱而不是 head 太小”是推断，不是某篇论文的直接定理。** 但这个推断和当前代码结构高度一致。

### 次要原因 4：defect support 的协议是妥协项，但不是当前首因

 现在的 `build_stage_b_split()` 是从 defect test 里抽 `support_defect`，再把它们从最终 `test_query` 中剔除。这个做法确实不是理想长期协议。

但从 最新状态看，shared split 与 shared scoring 已经统一，而核心症状依旧存在；A2 还表现出“pixel 有信号、image 不稳定”的特征。这说明**协议妥协是需要明确标注的二阶问题，但不是当前性能停滞的一阶原因。** 

### 次要原因 5：frozen DenseCLIP visual feature 可能不是最优，但目前不像首要瓶颈

 的 backbone 是 frozen DenseCLIP visual encoder，而且已经暴露了 `layer1~4 + global + local` 多层特征。

从结果看，A1 的 pixel AUROC 已经高于随机，A2 的 `defect-minus-normal` 还能把 pixel AUROC 拉到 `0.5728`。这说明**当前表征不是“完全死掉”的**，至少局部 patch 层面已经有一点异常分辨信号。更像是 scorer、prototype 和 correspondence 没有把这个信号用好。
这里也明确标一下：**“representation 不是首因”是推断**，不是已被实验彻底证明的事实。

## 2. 相关工作归纳

### A. 最贴近 当前路线、而且最适合单卡的工作

**PatchCore** 是最值得先借鉴的一个。它保留代表性的 nominal patch memory bank，而不是把正常 support 压成一个均值原型；在少样本条件下也很强，检测和定位都依赖 patch-level nearest neighbor。它最直接的意义就是：**先把 `feature_bank.py` 从“单均值”升级成“代表性 patch bank / kNN retrieval”**。这和 现有 frozen encoder 路线非常兼容，而且训练成本很低。([arXiv][2])

**WinCLIP** 证明了 CLIP 在 zero-/few-shot anomaly classification and segmentation 上是能工作的，而且 few-normal-shot 时只用少量正常图、无需 segmentation supervision；它的关键不是训练一个大模块，而是做 compositional prompt ensemble 和 window-based dense local scoring。最有价值的不是 prompt 本身，而是它强调了：**CLIP few-shot anomaly 的有效信号主要来自 dense local matching，不是单个全局中心。**([CVF开放获取][3])

**FastRecon** 和后续的 **FastRef** 也很适合 。FastRecon 的思路是：从 few normal support 出发，为 query 重建一个“正常版本”，再通过 sample alignment 找异常；FastRef 则进一步利用 query statistics 迭代修正 prototype，并用 OT 抑制 anomaly leakage，而且它是以“插件”方式加到 PatchCore / FastRecon / WinCLIP 等方法上的。这条线的价值在于：**在不做重训练的前提下，给 prototype 加 query-aware refinement。**([CVF开放获取][4])

**SubspaceAD** 是近期很值得重视的低成本路线。它主打 training-free 的 few-shot anomaly detection，用正常样本拟合低维子空间，再用重建残差做异常分数，不需要 prompt tuning，也不依赖大 memory bank。对单张 5060 来说，它非常适合作为“低成本强对照”。([arXiv][5])

### B. 对“image-level 与 pixel-level 不一致”最有启发的工作

**CLIP-AD** 最直接。它明确指出直接做 anomaly map 会出现 opposite predictions 和 irrelevant highlights，并把原因归到 joint space 不对齐上，然后通过 multi-level features 与 feature surgery 去修。这个问题描述，几乎就是现在 A2 指标冲突的文献版。最值得借它的，不是整套模型，而是两点：**多层特征融合** 和 **不要把 single-map 直接上升成 single image score**。([arXiv][1])

**Sea-CLIP** 进一步指出，很多 CLIP-based FSAD 忽略了 object semantics，导致 patch comparison 没有语义对应关系；它用 DINOv2 语义表征和 patch matching 解决这个问题。它的启示是：如果 memory bank / multi-prototype 还不够，那么下一步就不是 prompt，而是**correspondence / semantic matching**。但它因为引入第二个强 encoder，工程和显存开销都会上去，所以不适合做当前第一优先级。([CVF开放获取][6])

**Registration-based FSAD** 以及后续的 category-agnostic registration 路线，也说明 few-shot 工业异常里“先对齐、再比较”很重要。它们不是直接靠 prototype classification，而是把 registration 当成 proxy task，把 query 和 support normal 对齐后再找异常。当前如果后面走到 correspondence 分支，这条线会比 prompt 更应该优先。([arXiv][7])

### C. few-shot retrieval / dictionary 方向的近期进展

**DictAS** 把 few-shot anomaly segmentation 重写成 dictionary lookup，并用 contrastive query constraint 和 text alignment constraint 提高异常判别；**ReMP-AD** 则明确指出 few-shot memory bank 里不是所有 reference 都该等权，噪声和无关 reference 会拖垮 retrieval，所以它设计了 Intra-Class Token Retrieval 来抑制这种问题。这两条工作共同支持一个判断：**few-shot anomaly 的关键不是“有没有 prototype”，而是“reference 怎样保留、检索和加权”。**([CVF开放获取][8])

### D. Prompt / adapter / LoRA 路线：有价值，但不该现在上

如果未来要上 prompt，**PromptAD** 是最该先看的那个。它只用 normal samples 做 one-class prompt learning，并通过 anomaly margin 把 normal / anomaly prompt 拉开，属于 few-shot / low-data 场景下最贴近目标的一条 prompt 路线。它还用多层 vision-guided map 做融合，这也能给 scorer 设计带来启发。([CVF开放获取][9])

但 **AnomalyCLIP、AdaCLIP、AA-CLIP、VCP-CLIP** 这些路子，大多依赖 auxiliary anomaly data、object-agnostic prompt 训练、hybrid prompt 或两阶段视觉-文本对齐。它们很强，也很新，但对当前“单卡 5060 + 先把视觉 baseline 跑清楚”的阶段来说，工程复杂度偏高，不是最值当的下一步。([arXiv][10])

### E. 强但不适合当前硬件与工程阶段的路线

**One-to-Normal**、personalization / synthetic generation 这一类方法，会引入 anomaly-free generation、自定义生成模型、文本反演、synthetic data self-tuning 等额外模块。它们研究价值高，但对现在的目标来说太重了：当前最缺的不是生成能力，而是 scorer、prototype 和 correspondence 的基本判断。([OpenReview][11])

## 3. 下一阶段实验设计

### P0：先锁死 scorer，而不是继续加模块

**目标**：在当前 shared manifest 上，先找出一个稳定的 image-level scorer。
**为什么第一优先级是它**：现在最明显的症状就是 A2 的 sign 一换，image / pixel 指标就反向，这说明 scorer 比模型本体更先出问题。

**改哪些模块**

* `fewshot/scoring.py`
* `fewshot/stage_a1.py`
* `run_stage_a1.py`
* `run_stage_a2.py`
* 新增一个离线分析脚本，例如 `tools/eval_scoring_ablation.py`

**最小实验矩阵**

* 类别：先只做 `hazelnut`
* split：固定当前 shared manifest
* seed：`42, 43, 44`
* `support_normal_k = 8`
* `support_defect_k = 4`
* 固定一个 image size，建议先统一到 `320`

**要扫的 scorer**

* A1：`1 - normal_map`、`-normal_map`
* A2：`defect - normal`、`normal - defect`、`blend`
* aggregation：`patch_max`、`patch_topk_mean@1%`、`@5%`、`@10%`
* aggregation 位置：`patch map` 与 `upsampled map` 分开试
* 额外建议：利用现有 encoder 已经输出的 `layer3 / layer4 / local`，做**同一 encoder 的两层融合 scorer**，因为这是低成本借 CLIP-AD / PromptAD 的方式，不需要新模型。 ([arXiv][1])

**验收标准**

* 3 个 seed 下，`mean(score_defect) > mean(score_good)` 方向一致
* image AUROC 至少比当前最好结果高 `0.03`
* pixel AUROC 不显著退化
* 选出一套默认 scorer，以后 A1/A2/B 全部共用

### P1：以 P0 锁定的两条 scorer 分支为锚点，验证 retrieval / multi-prototype 是否击败单均值 prototype

**目标**：在不再混淆 scorer 与 prototype family 的前提下，验证当前瓶颈是不是 prototype 太粗。  
**这是下一步主线。**

先把 P0 的两个锚点固定下来：

* **主线锚点**：`A1 + layer4 + neg-normal + max + patch`
  当前 3-seed 基线：`image 0.5857 +/- 0.0216`，`pixel 0.4976 +/- 0.0472`
* **pixel-first 对照锚点**：`A2 + layer4 + normal-minus-defect + topk_mean + patch + topk=0.10`
  当前 3-seed 基线：`image 0.5064 +/- 0.0305`，`pixel 0.5539 +/- 0.1272`

这里有一个重要约束：**不要把 “A1 只配 memory bank / kNN” 与 “A2 只配 multi-prototype” 绑死。**  
否则后面如果结果变好，无法判断提升到底来自 scorer 分支，还是来自 prototype family。  
所以 P1 应该是 **两条 scorer 分支固定，prototype family 共用同一套实验矩阵**。

**改哪些模块**

* `fewshot/feature_bank.py`
* `fewshot/scoring.py`
* 新增 `fewshot/retrieval.py` 或 `fewshot/prototypes.py`
* `run_stage_a1.py`
* `run_stage_a2.py`

**P1 第一轮只做两类 prototype family**

1. **PatchCore-like support memory bank + kNN**
2. **multi-prototype**：对 normal / defect support patch 做聚类，保留 `K=4/8` 个 prototype

`FastRef-lite` 暂时不放进 P1 第一轮。原因不是它没价值，而是它会把 “query-aware refinement” 这个新变量也一起引进来；先让 `memory bank / kNN` 和 `multi-prototype` 直接分胜负，更容易判断问题。

**最小实验矩阵**

* `hazelnut`
* 主线分支：`A1 + layer4 + neg-normal + max + patch`
* 对照分支：`A2 + layer4 + normal-minus-defect + topk_mean + patch + topk=0.10`
* prototype family：`memory bank + kNN` vs `multi-prototype(K=4/8)`
* 第一轮先固定 `support_normal_k = 8`，`support_defect_k = 4`
* seed：`42, 43, 44`

说明：`support size` 不应在 P1 第一轮和 prototype family 一起大扫。  
先固定在 P0 的 `8/4`，把变量收缩到“prototype 怎么表示”本身；等 P1 出现明确 winner，再做 `support size` 扩展更干净。

**验收标准**

* **主线成功标准**：`image AUROC mean` 要稳定超过当前 A1 default，即高于 `0.5857`，并且 3 个 seed 里至少 2 个 seed 分别超过各自的 A1 基线
* **主线保底约束**：`pixel AUROC mean` 不能明显跌破当前 A1 default，即不能比 `0.4976` 再明显下滑
* **pixel-first 对照成功标准**：若要继续保留 A2 分支，`pixel AUROC mean` 应尽量不低于当前 pixel-first 基线 `0.5539`；若均值略低，也至少要显著降低当前的大波动（当前 `pixel std = 0.1272`）
* **共用标准**：balanced 指标不能比当前 A1 default 更差太多；若 image 提升但 pixel 明显塌缩，该配置不晋级下一轮
* 三个 seed 中至少两个 seed 方向一致

### P2：如果 P1 还不够，再上 correspondence / subspace，而不是 prompt

**目标**：解决“patch 有信号，但 image 不稳定”的第二层问题。

我建议这里优先两条：

**方案 A：SubspaceAD-style normal subspace residual**
它是 training-free、很省资源，而且能作为一个与 prototype 完全不同的 low-cost 对照。如果它明显优于 prototype 路线，就说明问题更多出在 prototype 假设，而不是 DenseCLIP backbone 本身。([arXiv][5])

**方案 B：lightweight correspondence / registration**
先用同一个 DenseCLIP encoder 做局部最近邻配对或受空间约束的 patch matching；只有这条也不够时，才考虑 Sea-CLIP 那种更重的语义 encoder。([arXiv][7])

**改哪些模块**

* 新增 `fewshot/subspace.py`
* 新增 `fewshot/matching.py`
* `fewshot/scoring.py`
* `run_stage_a1.py`
* `run_stage_a2.py`

**验收标准**

* image / pixel 指标不再出现大幅对冲
* PRO 开始纳入报告，因为协议本来就要求 region-level metric 尽量补齐

### P3：只有在视觉 baseline 稳定后，才重新评估 B

**目标**：判断 tiny head 是否真有增益。
**这里不要再做“多加 epoch 看看”。**

**改哪些模块**

* `fewshot/head.py`
* `fewshot/learned_head.py`
* `run_stage_b.py`
* 新增 `fewshot/losses.py`

**先做的改动**

1. 增加一个**线性 head**：单层 `1x1 conv`，作为比当前 MLP 更稳定的控制组
2. 去掉或先关掉 dropout
3. 用 `pixel BCE + Dice` 为主损失，`image BCE` 只做辅损失
4. 训练日志增加：

   * train image AUROC
   * patch Dice / IoU
   * image score 方差
     因为现在最大的问题之一就是“接近常数输出”，这必须直接监控。  

**最小实验矩阵**

* `hazelnut`
* 使用 P1/P2 里最强的 similarity/scorer，不再用旧单均值
* head：`linear` vs `current_mlp`
* loss：`image BCE` vs `pixel BCE + Dice + 0.25 image BCE`
* seed：`42, 43, 44`

**验收标准**

* 不再塌缩：image score 方差明显大于近常数
* train pixel 指标有实质提升
* query 上 B 至少稳定超过最强 A2；若做不到，就把 B 降级为次线，不再优先投入

### P4：满足下面条件后，再扩全 15 类

只在满足这些条件后扩：

* 默认 scorer 已锁定
* single-mean prototype 已被 retrieval / subspace / correspondence 中至少一条稳定击败
* B 是否值得保留已经判清
* 报告里补上 PRO、runtime、support size、resolution 等元数据

扩展顺序仍按协议：

* 先单 seed 跑全 15 类
* 再只对 finalist 配置做 3 seeds
* 单 GPU 顺序执行，不并发

## 4. 路线决策建议

### 1）下一步是否应继续以 A1/A2 为主，而不是提前上 prompt learning？

**是。继续 A1/A2。不要提前上 prompt。**

原因有两个。
第一，协议已经明确把 prompt 放到 visual baseline 之后。
第二，当前结果最像 scorer / prototype / correspondence 问题，不像“必须靠语言先验才能动”的问题；而且 PromptAD 这类工作虽然适合 few-shot normal-only 场景，但依然是下一阶段，而不是现在。([CVF开放获取][9])

### 2）B 路线如果继续，应该先改 loss、supervision，还是 prototype？

**全局上，先改 prototype / scorer；一旦进入 B 分支，先改 supervision / loss。**

更具体地说：

* **对整个项目主线**：prototype / scorer 优先，因为它们现在同时限制 A1/A2/B
* **对 B 这个分支内部**：supervision / loss 优先，因为当前 B 基本没在用 mask 信息，只做 image BCE，太弱了  

### 3）是否应优先尝试 multi-prototype / memory bank / patch retrieval，而不是单均值 prototype？

**是，强烈建议优先。**

这是我最明确的一条建议。PatchCore、DictAS、ReMP-AD、FastRef 这一串工作的共同结论就是：few-shot anomaly detection 的参考集不该被压成单一中心，而应该保留 representative / retrievable / refinable 的 patch-level 结构。 `feature_bank.py` 正好是这件事的最薄弱处。 ([arXiv][2])

### 4）在单张 5060 条件下，最值得优先尝试的 3 条低成本路线是什么？

按优先级排：

**第一条：A1/A2 + PatchCore-like memory bank / multi-prototype / kNN retrieval**
这是与现有代码最兼容、训练成本最低、文献支持最强的一条。([arXiv][2])

**第二条：A1/A2 + same-encoder multi-scale scorer + FastRef-lite query refinement**
因为 backbone 已经吐出多层特征，这条路线几乎不增加模型规模，但很可能直接缓解 image/pixel 不一致。 ([arXiv][1])

**第三条：SubspaceAD-style normal subspace residual**
这是 training-free 的强对照，特别适合单卡和低工程成本。它能帮助判断：问题主要在 prototype，还是在 encoder 表征本身。([arXiv][5])

补一句：
如果以后真的要上 prompt，我会把 **PromptAD** 放在第一候选，而不是先上 AdaCLIP / AnomalyCLIP / AA-CLIP 这类更重的路线。([CVF开放获取][9])

---

最后一句总结：

**当前最该做的不是“让模型更会学”，而是“让 anomaly signal 先被正确表示、检索和聚合”。**
对这个项目来说，下一阶段最值当的顺序是：

**shared scorer 定型 → 单均值 prototype 升级为 retrieval / multi-prototype / subspace → 必要时补 lightweight correspondence → 最后才决定 B 和 prompt。**

我可以下一条直接把上面的 P0–P4 改写成一份**可执行实验清单**，按“脚本 / 参数 / 输出目录 / 判定门槛”展开。
