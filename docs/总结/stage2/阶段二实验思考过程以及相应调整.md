## 1. P0 之后已经可以确定的事实

P0 用 `hazelnut`、固定 shared manifest、`seed = 42 / 43 / 44`，完成了 scorer / aggregation / feature layer 的完整 sweep。  
正式结果以 `outputs/stage2/p0_scoring/final_aggregate` 为准。

已经确定的事实有四条：

### 1.1 当前最稳的默认主线是 A1，不是 A2

P0 的 3-seed 聚合结果表明，当前最稳的主线配置是：

- `A1 + layer4 + neg-normal + max + patch`

对应均值为：

- `image AUROC = 0.5857 +/- 0.0216`
- `pixel AUROC = 0.4976 +/- 0.0472`

这组结果不是 pixel 最强，但它在 image 和 balanced 上都最好，而且波动明显比 A2 的 pixel-first 配置更小。

### 1.2 A2 的 pixel-first scorer 可以保留，但不能直接升为默认

P0 中 pixel AUROC 最强的是：

- `A2 + layer4 + normal-minus-defect + topk_mean + patch + topk=0.10`

对应均值为：

- `image AUROC = 0.5064 +/- 0.0305`
- `pixel AUROC = 0.5539 +/- 0.1272`

问题不在于它不强，而在于它**不够稳**。  
pixel 标准差达到 `0.1272`，说明它很可能更适合作为“pixel-first 上界对照”，而不是当前 Stage 2 的默认主线。

### 1.3 P0 的主要信号来自 feature layer，而不是 aggregation 小参数

P0 的 sweep 结果里，大量配置在 `aggregation_mode / aggregation_stage / topk_ratio` 上直接打平或接近打平。  
真正持续拉开差距的是：

- `feature_layer = layer4` vs `local`
- `A1` vs `A2`
- `normal-only scorer` vs `dual-prototype scorer`

这意味着 P1 不应再继续扩大 scorer 搜索空间，而应把注意力转向 prototype family。

### 1.4 blend 目前没有继续投入价值

`blend` 在 P0 里没有显示出独立增益，整体表现明显落后于：

- A1 默认主线
- A2 pixel-first 对照

因此后续若实验预算紧张，`blend` 可以直接降级，不再作为优先扫描项。

## 2. 为什么 P1 不能写成“主线=A1+kNN，对照=A2+multi-prototype”

这版想法直觉上合理，但严格来说不够干净。

问题在于：它同时改了两件事：

1. scorer 分支不同
2. prototype family 也不同

如果最后结果变好，就无法回答一个关键问题：

- 提升到底来自 `A1` 相比 `A2`
- 还是来自 `memory bank / kNN` 相比 `multi-prototype`

这会直接削弱 P1 的解释力。

所以 P1 的正确写法应该是：

- 先固定两条 scorer 锚点
- 再让这两条 scorer 分支共用同一套 prototype family 实验矩阵

也就是说，P1 不是：

- `A1 -> kNN`
- `A2 -> multi-prototype`

而应该是：

- `A1 anchor -> kNN / multi-prototype`
- `A2 anchor -> kNN / multi-prototype`

这样 P1 才能真正回答：

- 单均值 prototype 的瓶颈是否存在
- 更好的 family 是 `memory bank + kNN` 还是 `multi-prototype`
- 这类提升在 A1 和 A2 上是否一致

---

## 3. 为什么 P1 第一轮不应同时扫 support size

P1 的核心问题是：

- “prototype 该怎么表示”

而不是：

- “support sample 数量对结果的影响有多大”

如果第一轮同时扫：

- `support_normal_k = 4 / 8 / 16`
- `support_defect_k = 1 / 4`
- `prototype family = kNN / multi-prototype`

那么变量会一下子变成三组：

1. scorer 分支
2. prototype family
3. support size

在 `hazelnut` 单类阶段，这样会让结论过早变脏。

因此更合理的顺序是：

### 第一轮

固定 `support_normal_k = 8`、`support_defect_k = 4`，只比较：

- `memory bank + kNN`
- `multi-prototype(K=4/8)`

### 第二轮

等第一轮出现 clear winner 之后，再对 winner 扫：

- `support_normal_k = 4 / 8 / 16`
- `support_defect_k = 1 / 4`

这样每一轮回答的问题都更清晰。

---

## 4. 为什么 FastRef-lite 不应放进 P1 第一轮

`FastRef-lite` 本身是值得尝试的，但它会引入新的变量：

- query-aware refinement
- prototype 不是静态的，而是带 query 反馈的

这会把问题从：

- “prototype 表示方式是否不够好”

直接扩大成：

- “prototype 表示方式”
- “query-aware refinement 是否必要”

在 P1 第一轮里，这个变量太早了。  
更稳的顺序应该是：

1. 先比较静态 family：`kNN` vs `multi-prototype`
2. 如果静态 family 还不够，再引入 `FastRef-lite`

这样 `FastRef-lite` 更适合作为：

- P1 第二轮
- 或 P2 的轻量 refinement 分支

而不是 P1 的第一批实验。

## 5. P1 现阶段应固定的两条锚点

P1 目前应固定两条 scorer 锚点：

### 5.1 主线锚点

- `A1 + layer4 + neg-normal + max + patch`

用途：

- 作为 Stage 2 当前默认主线
- 判断 retrieval / multi-prototype 能否稳定提高 image-level 结果

基线：

- `image AUROC = 0.5857 +/- 0.0216`
- `pixel AUROC = 0.4976 +/- 0.0472`

### 5.2 pixel-first 对照锚点

- `A2 + layer4 + normal-minus-defect + topk_mean + patch + topk=0.10`

用途：

- 作为 pixel 优先对照
- 判断 retrieval / multi-prototype 能否在不牺牲过多 image 的前提下保住 pixel 优势

基线：

- `image AUROC = 0.5064 +/- 0.0305`
- `pixel AUROC = 0.5539 +/- 0.1272`

## 6. P1 的调整后实验框架

### 6.1 第一轮只做两类 prototype family

#### family A

- `PatchCore-like memory bank + kNN`

#### family B

- `multi-prototype`
- 对 normal / defect support patch 聚类
- 先做 `K = 4 / 8`

### 6.2 两条 scorer 分支都跑同一套 family

具体来说，P1 第一轮应是：

- `A1 anchor + memory bank + kNN`
- `A1 anchor + multi-prototype(K=4)`
- `A1 anchor + multi-prototype(K=8)`
- `A2 anchor + memory bank + kNN`
- `A2 anchor + multi-prototype(K=4)`
- `A2 anchor + multi-prototype(K=8)`

### 6.3 先固定 support size

P1 第一轮先固定：

- `support_normal_k = 8`
- `support_defect_k = 4`

不在这一轮扩 support size。

## 7. P1 的成功标准应该怎么写

原来的“image 提升、pixel 不明显下降”这个方向是对的，但必须量化。

### 7.1 主线成功标准

对于 A1 主线：

- `image AUROC mean` 要高于当前默认主线 `0.5857`
- 3 个 seed 中至少 2 个 seed 分别超过各自 A1 基线

### 7.2 主线保底约束

对于 A1 主线：

- `pixel AUROC mean` 不能明显跌破当前默认主线 `0.4976`

这里故意不要求它必须超过 `0.5539`，因为那是 A2 pixel-first 路线的目标，不应强绑到主线上。

### 7.3 pixel-first 对照的保留条件

对于 A2 对照：

- `pixel AUROC mean` 应尽量不低于当前 `0.5539`
- 如果均值略低，也至少要显著降低当前大波动

也就是说，A2 分支未来是否保留，不只看均值，还看稳定性是否改善。

### 7.4 共用标准

无论 A1 还是 A2：

- `balanced` 不能比当前 A1 默认主线更差太多
- 如果 image 提升但 pixel 明显塌缩，该配置不晋级
- 3 个 seed 中至少 2 个 seed 的方向要一致

---

## 8. 对 P2-P4 的连锁影响

### 8.1 对 P2

如果 P1 证明：

- `memory bank + kNN`
- `multi-prototype`

都无法同时稳住 image 和 pixel，那么下一步就不是继续扫 scorer，而是进入：

- `subspace residual`
- `lightweight correspondence / matching`

### 8.2 对 P3

只有在 P1/P2 先把视觉 baseline 做稳之后，才值得重新评估 B。  
否则 B 学到的仍然会是一个建立在不稳定 similarity / scorer 上的弱监督头。

### 8.3 对 P4

在扩到 15 类之前，至少应满足：

- 默认 scorer 已锁定
- P1 至少找出一个比单均值 prototype 更稳的 family
- 是否保留 A2 pixel-first 分支已经判清

---

## 9. 当前调整后的结论

一句话总结这轮调整：

**P1 不是“选 A1 还是 A2”，而是“在已锁定的 A1/A2 锚点上，判断哪种 prototype family 值得继续投”。**

因此，接下来的最合理顺序是：

1. 固定两条 P0 scorer 锚点
2. 比较 `memory bank + kNN` 与 `multi-prototype`
3. winner 出来后再扩 support size
4. 若仍不够，再进入 `FastRef-lite / subspace / correspondence`

这比继续放大 scorer sweep，更符合 P0 已经给出的信号。

### 10.2 截至 P1 结束后，顺序上发生了什么变化

P1 已经把“prototype family 谁值得继续投”这个问题回答清楚了。  
结合正式结果文档，当前可以把后续顺序收敛为：

1. 以 A1 主线上的 `memory_bank` 作为唯一视觉主线继续推进
2. P2 不再比较 `memory_bank` vs `multi-prototype`，而是直接补 `subspace / correspondence / FastRef-lite`
3. A2 不再作为并行主线推进，只保留为历史对照
4. B 之后若继续评估，也应复用 P1/P2 的 winner，而不是回退到单均值 prototype

### 10.3 当前这份文档和结果文档的分工

到现在为止，两份文档的职责应明确区分：

- `阶段二结果.md`：写正式结果、winner、aggregate 路径和数值
- 本文：保留判断链条，并在阶段结束后补一段“现在顺序怎么变了”

这样后面即使继续进入 P2/P3，也能同时保留：

- 当时为什么这样设计
- 最后结果把顺序改成了什么

### 10.4 基于 P1 正式结果，调整后的 P2 计划

P1 收口后，P2 的起点已经不能再写成宽泛的“继续补 retrieval / subspace / correspondence”。  
现在唯一合理的写法应该是：

- **固定主线**：`A1 + layer4 + neg-normal + max + patch + memory_bank + support_normal_k=16 + reference_topk=3`
- **固定数据范围**：先只做 `hazelnut`
- **固定评估口径**：正式结论仍以 3 个 seed 聚合为准，但新的 P2 变体可以先从 `seed42` smoke 开始
- **当前代码落地范围**：先只做工程侧的 `support` 图预加载到内存，不把 `FastRef-lite / correspondence / subspace` 和工程提速项混在同一轮代码提交里

也就是说，P2 不再回答：

- `memory_bank` 和 `multi-prototype` 谁更好
- `A1` 和 `A2` 谁值得继续投
- `support_normal_k` 还要不要继续扫

这些问题都已经在 P1 里回答过了。  
P2 现在只回答一个新问题：

- **在已经成立的 `A1 + memory_bank` 主线上，哪种“结构增强”能进一步提高 image-level 判别，同时不破坏已经很强的 pixel-level 表现。**

#### P2 的顺序应当收窄成两步，而不是并行开很多新分支

##### 第一步：先完成最小工程落地，再做 retrieval 上的轻量 refinement

这一步先拆成两个层次，避免把工程改动和方法改动绑在一起：

1. **工程层**：只落地 `support` 图预加载到内存
2. **方法层**：在工程侧稳定后，再开始 `FastRef-lite / lightweight correspondence`

优先级最高的不是 prompt，也不是 learned head，而是围绕当前 `memory_bank` winner 做最小结构增强：

1. `FastRef-lite`
2. `lightweight correspondence / matching`

这一步的约束应该写死：

- 不换 encoder
- 不引入第二个大模型
- 不引入文本分支
- 不训练 backbone
- 尽量保持训练自由或近训练自由

目标是判断：

- query-aware retrieval/refinement 能否把 `memory_bank` 的 reference 用得更干净
- 是否能在不牺牲 pixel 的前提下，把 image AUROC 再抬高一点

##### 第二步：如果第一步没有稳定增益，再做 subspace residual

`subspace` 在当前阶段更适合作为第二顺位，而不是和 refinement 同时平铺。原因是：

- P1 已经证明 reference bank 有效
- 所以 P2 先该问“retrieval 还能不能再变好”
- 只有当 retrieval refinement 不能稳定带来提升时，才需要把问题改写成“局部正常子空间残差是否比直接相似度更好”

因此 P2 的执行顺序应为：

1. `A1 + memory_bank + FastRef-lite`
2. `A1 + memory_bank + lightweight correspondence`
3. 如果前两者都没有稳定增益，再做 `A1 + memory_bank + local subspace residual`

而不是一开始就把三条线并行放大。

#### P2 不再保留哪些旧变量

从 P1 结束开始，下面这些变量都不应再进入 P2 主实验矩阵：

- `A2` 主线
- `multi_prototype`
- `mean prototype`
- 新一轮大规模 scorer sweep
- `support_normal_k = 4 / 8 / 16` 全扫
- `reference_topk = 1 / 3 / 5` 全扫

这些变量只在两个场景下保留：

- 作为必要回退对照
- 或者在最终 winner 确认后做一次很小规模 sanity check

#### P2 的默认对照组应当怎么写

P2 至少保留两个固定对照，不能只看新方法自己的数字：

1. `P1 winner`
   - `A1 + layer4 + neg-normal + max + patch + memory_bank + sn16 + refk3`
   - 基线：`image 0.6732 +/- 0.0050`，`pixel 0.9535 +/- 0.0071`

2. `P1 近邻回退`
   - `A1 + layer4 + neg-normal + max + patch + memory_bank + sn16 + refk1`
   - 用于判断 `P2` 的提升是否只是把 `refk` 换了个写法

如果某个 P2 变体连这两个固定对照都过不去，就不应再往后扩。

#### P2 的成功标准需要比 P1 更严格

因为 P1 的基线已经很强，P2 不应该再用“略有提升就算成功”的宽松标准。  
更合理的门槛是：

1. `image_auroc_mean` 要稳定高于 `0.6732`，或者
2. `balanced_mean` 要稳定高于 `0.8134`
3. 同时 `pixel_auroc_mean` 不应明显低于 `0.9535`
4. `image/std` 和 `pixel/std` 不应明显恶化

如果只是：

- image 略涨一点，但 std 变大
- 或 pixel 明显回落

这种配置不应晋级为 P2 winner。

#### P2 的最小落地版本

如果只做一个最小可执行版本，顺序应当是：

1. 在当前 `memory_bank + sn16 + refk3` 主线上先落地 `support` 图预加载到内存
2. 工程改动完成后，若继续做方法实验，先实现 `FastRef-lite`
3. 新方法先跑 `hazelnut / seed42`
4. 若单 seed 有正向信号，再补 `43 / 44`
5. 若 3-seed 仍有增益，再保留为 P2 主线
6. 若没有增益，再切到 `lightweight correspondence`

这样做的好处是：

- 保持变量最少
- 不会把 P2 重新做成一个新一轮大扫参
- 也更符合当前已经很强的 `P1` baseline

#### 当前这一轮调整后的结论

基于 P1 正式结果，P2 的正确写法已经收敛为：

- **唯一主线**：`A1 + memory_bank + sn16 + refk3`
- **唯一目标**：在不破坏强 pixel 表现的前提下，继续提高 image-level 判别与跨类稳定性
- **第一优先级**：retrieval refinement
- **第二优先级**：subspace residual
- **暂不优先**：A2、multi-prototype、B、prompt

## 11. P2 收口后的补记

这部分只补记 P2 跑完后，哪些判断被证实、哪些地方需要收窄口径；不回写前面的设计过程。

### 11.1 工程侧最小改动是成立的

P2 之前先落地了 few-shot 场景下的 `support` 图预加载到内存。  
这一步没有改变数值逻辑，只是减少了反复读图与构 batch 的开销；控制组结果仍与 P1 baseline 处于同一量级。

这说明：

- 这个工程改动可以保留并继续复用到 P3/P4
- 后面如果继续做结构增强，不需要再回头担心 support 侧 I/O 抖动

### 11.2 P2 的“先 refinement，后 subspace”顺序在执行后被进一步校正

按照设计时的判断，P2 优先顺序是：

1. `FastRef-lite`
2. `lightweight correspondence / matching`
3. `subspace`

但实际跑完后，结果并不是按这个顺序成功：

- `FastRef-lite` 在 smoke 中几乎与 baseline 重合，没有形成独立增益
- `matching` 在单 seed 上有局部信号，但补齐 3-seed 后没有稳定超过 baseline
- `subspace(dim=8)` 反而成为唯一一个在均值上同时抬高 image、pixel 和 PRO 的候选

这说明 P2 的方法优先级虽然按“先 refinement 再 subspace”展开是合理的，但**结果层面真正留下来的只有 subspace 分支**。

### 11.3 为什么 P2 不能直接宣布 subspace 接管默认主线

如果只看 3-seed 均值，`subspace(dim=8)` 的确优于 baseline。  
但 P2 在设计时就写过一个更严格的门槛：

- 不只看均值
- 还要看 `image/std` 与 `pixel/std` 是否明显恶化

实际结果里：

- baseline：`image 0.6732 +/- 0.0050`
- subspace(dim=8)：`image 0.6740 +/- 0.0249`

也就是说，image 侧均值只涨了一点，但波动放大得很明显。  
所以更严谨的判断应是：

- `subspace(dim=8)` 是 P2 唯一保留候选
- 但它还不够稳定，不能直接升级成新的默认线

### 11.4 P2 跑完后，后续顺序应怎样写

到 P2 收口时，更合理的后续顺序应变成：

1. 默认主线继续保持 `A1 + memory_bank + sn16 + refk3`
2. `subspace(dim=8)` 作为唯一保留增强分支进入后续验证
3. `FastRef-lite` 与 `matching` 暂不继续扩大投入
4. 如果后续做跨类或更大规模验证，应优先回答 `subspace(dim=8)` 的稳定性问题，而不是重新回到 refinement 大扫参

### 11.5 到这一步，P2 真正回答了什么

P2 最终真正回答的不是“哪种结构增强均值最高”，而是两件更具体的事：

1. 在当前 `A1 + memory_bank` 强 baseline 上，`FastRef-lite` 和 `matching` 还不足以形成可复用的新主线
2. `subspace(dim=8)` 证明了结构增强仍有继续提升空间，但稳定性还不够，不应仓促扶正

因此，P2 的收口方式不应写成“新 winner 已出现”，而应写成：

- **默认主线保持不变**
- **新增一个值得继续验证的保留分支：subspace(dim=8)**

## 12. P4 收口后的补记

这部分只补记 P4 跑完后，扩到全 15 类之后到底发生了什么；不回写前面 P2 收口时的判断顺序。

### 12.1 为什么这轮 P4 只带视觉路线，不带 B

设计文档里 P4 的前置条件包含“P3 已经明确 B 是否保留”。  
但当前仓库里并没有 B 的正式 finalist 归档，所以这轮真正可执行的 P4 只能先落到视觉路线：

- `P1 baseline`
- `P2 保留分支 subspace(dim=8)`

也就是说，这轮 P4 真正回答的是：

- **扩到 15 类时，当前视觉路线该带哪一个 finalist 继续走**

而不是：

- “视觉”和 “B” 谁更值得进入下一阶段

### 12.2 P4 真正改变了什么判断

P2 收口时，对 `subspace(dim=8)` 的判断还是：

- 在 `hazelnut` 上均值略优
- 但 image 波动更大
- 因此只能保留为候选，不能扶正

P4 跑完后，这个判断发生了关键变化：  
一旦扩到全 15 类，`subspace(dim=8)` 在 `seed42` screening 上直接整体压过了纯 baseline，说明它不是只在 `hazelnut` 上偶然有效。

也就是说，P4 证明了：

- `subspace(dim=8)` 的价值不是单类幻觉
- 它确实值得成为 full-15 口径下的默认视觉 finalist

### 12.3 为什么即使 P4 winner 已经明确，也不能把问题写成“Stage 2 已经稳定”

P4 给出了很清楚的 winner，但也同时暴露了新的事实：

- 强类非常强，例如 `bottle / toothbrush / transistor`
- 弱类仍然明显偏弱，例如 `leather / grid / carpet / screw`

这说明现在最准确的描述不是：

- “15 类 baseline 已经稳定”

而是：

- “15 类该带哪条视觉线继续走，已经判清”
- “但这条线在不同类别上的稳定性仍然不均匀”

因此，P4 的收口不能写成大团圆，而应写成：

- **winner 已经确定**
- **但 failure pattern 也已经更清楚地暴露出来了**

### 12.4 P4 跑完后，后续真正该优先看的不再是 scorer，而是失败类模式

到这一步，继续回去放大 scorer sweep 或 refinement 小参数的收益已经更可疑了。  
因为 P4 暴露出来的主要问题不再是“默认 scorer 选错了”，而是：

- 某些类别本身对当前视觉表示更不友好
- 类别间表现差异远大于同类内部的小参数差异

所以如果后面继续推进，更合理的关注点应是：

1. `leather / grid / carpet / screw` 这类弱类到底是 retrieval 问题、subspace 问题，还是表征问题
2. 是否需要按类分析 failure mode，而不是再做一轮统一大扫参
3. 如果后续重新评估 B 或下一阶段方法，应默认从 `subspace(dim=8)` 这条 full-15 winner 起步

### 12.5 到 P4 为止，Stage 2 视觉路线的收敛轨迹已经很清楚

如果把 P0 到 P4 串起来，视觉路线的收敛顺序已经固定了：

1. P0 锁定默认 scorer：`A1 + layer4 + neg-normal + max + patch`
2. P1 证明单均值 prototype 是主瓶颈，winner 变成 `memory_bank + sn16 + refk3`
3. P2 证明 `subspace(dim=8)` 是唯一值得保留的结构增强分支
4. P4 进一步证明：扩到全 15 类后，真正应保留的视觉 finalist 就是 `subspace(dim=8)`，而不是回退到纯 baseline

这条轨迹本身已经说明：

- 视觉路线的关键增益主要来自 `prototype representation -> structure enhancement`
- 而不是来自后面再补 scorer 小技巧
